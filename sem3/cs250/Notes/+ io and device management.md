# + io and device management
> How the processor handles unexpected events and communicates with devices

---
## Exceptions

> An unscheduled event that disrupts normal instruction execution; control transfers to an OS handler.

**Interrupt** — exception originating *outside* the processor (hardware device, timer)

### Common Causes
- Arithmetic errors (overflow, divide-by-zero)
- Illegal / undefined instructions
- Memory violations (page fault, segfault)
- I/O device signals
- System calls (intentional, synchronous)

### Classification

By source:
- **External (Interrupt)** — generated by hardware outside CPU (I/O completion, timer)
- **Internal (Trap/Fault)** — generated by the executing instruction (overflow, page fault)

By timing:
- **Synchronous** — caused by the current instruction (divide-by-zero, syscall)
- **Asynchronous** — independent of instruction execution (I/O interrupt, clock interrupt)

### Exception Handling Steps
1. Stop normal instruction execution
2. Save processor state (PC, status registers → ELR, ESR on ARMv8)
3. Identify the cause
4. Transfer control to handler (OS exception handler)
5. Handle the exception
6. Resume execution (if possible) or terminate

**Vectored interrupt** — handler address determined by exception cause; faster dispatch
**Restartable instruction** — can resume after the exception is handled (e.g., page faults, cache misses)


---
## Exceptions in Pipelined Processors

Exceptions are treated as a **control hazard**:

**Precise exception** — all instructions before the faulting instruction have completed; no later instructions have modified state; required by most modern ISAs

**Imprecise exception** — harder to handle; faulting instruction is ambiguous

Out-of-order processors maintain precise exceptions via:
- **Reorder buffer (ROB)** — holds results until safe to commit
- **In-order commit** — results written to visible state in program order
- **Register renaming** — eliminates false dependencies

## Device Programming

How the CPU communicates with I/O devices:

### Memory-Mapped I/O
> Device registers are mapped into the normal address space.
- CPU reads/writes device registers using standard load/store instructions
- Simpler; no special instructions needed
- Common in RISC architectures (ARM, RISC-V)

### Port-Mapped I/O
> Separate address space for I/O devices (x86 `in`/`out` instructions).
- Requires special I/O instructions

### Polling
> CPU continuously reads a device status register in a loop until the device is ready.
- Simple implementation
- Wastes CPU cycles if device is slow

### Interrupts
> Device signals the CPU when it needs attention; CPU suspends current task and runs handler.
- CPU is free to do other work while device operates
- More efficient than polling for slow or infrequent events
- Uses the exception mechanism

Interrupt-driven I/O flow:
1. CPU starts I/O operation
2. CPU continues executing other instructions
3. Device completes, asserts interrupt signal
4. CPU finishes current instruction, saves state
5. CPU jumps to interrupt handler
6. Handler reads/writes device data
7. CPU restores state and resumes

See [[Device Programming]]

## DMA (Direct Memory Access)

> A hardware controller that transfers data directly between an I/O device and main memory, without CPU involvement in each transfer.

Benefits:
- Frees CPU to do other work during large data transfers
- Higher throughput for high-speed devices (disk, network, GPU)

### DMA Transfer Process
1. CPU programs DMA controller: source, destination, transfer size
2. DMA controller manages the transfer bus cycle by bus cycle
3. DMA interrupts CPU when transfer is complete
4. CPU handles any needed processing

Used for: disk reads/writes, network packets, GPU memory transfers

See [[DMA (Direct Memory Access)]]

---
## CPU Time vs Execution Time

**CPU Time** — time spent executing instructions (excludes I/O waits)
**Execution Time (wall-clock time)** — full time from start to finish, including I/O waits

I/O-bound programs spend most time waiting on devices; CPU time << execution time.

---
## External Devices

**GPU (Graphics Processing Unit)**
- Originally for rendering; now used for general-purpose parallel computing (GPGPU)
- Thousands of simpler cores organized as SIMT (Single Instruction Multiple Thread)
- Very high memory bandwidth
- Programmed via CUDA (NVIDIA) or OpenCL

**Network**
- **LAN** — local area network (building scale)
- **WAN** — wide area network (continent scale)
- **Network bandwidth** — peak transfer rate of a link or collective network

See [[External Devices]], [[Network]]

# Active Recall

1. What is the difference between polling and interrupts?
||Polling: CPU loops checking device status (wastes cycles). Interrupts: device signals CPU when ready; CPU can do other work in between.||

2. What does DMA do and why is it useful?
||DMA transfers data between a device and memory without CPU involvement in each cycle, freeing the CPU to do other work and achieving higher throughput.||

3. What is a precise exception?
||One where all earlier instructions have completed and no later instructions have modified architectural state — the faulting instruction is exactly identified.||
